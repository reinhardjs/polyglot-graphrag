## 1. Architecture Principles (what makes this "enterprise-grade")

These are non-negotiable for v1.0. Every phase must advance toward these:

1. **Domain-agnostic core** — add a domain by writing `domain_config.yaml` + a folder with `retrieve.py` and `ingest.py`. Nothing in `serve_gpu.py` or `federated.py` is domain-specific.

2. **Graceful degradation** — missing domain, dead Neo4j, empty Qdrant collection, or failed E4B synthesis must never 500. Return partial results with `"degraded": true` and a `"notice"` explaining what was unavailable.

3. **One request, one response** — `/ask` is the only endpoint a consumer needs. Domain, mode, synthesis, confidence floor, and dual-evidence preference are parameters on the same call.

4. **Provenance in every result** — every context in `/ask` carries `signal`, `concept_id`, `confidence`, `doc_id`, `chunk_idx`. No context without a source.

5. **Try-first documentation** — every domain ships with a QUICKSTART that includes copy-pasteable curl commands. A user new to the system runs one command and gets a result.

6. **Horizontal scale for retrieval** — retrieval (embed + search) must be parallelizable across domains. The federation layer runs retrievers concurrently; adding domains adds parallelism, not latency.

7. **Deterministic before LLM** — every inferred relationship must first be attempted via deterministic graph analysis (Cypher, topology, rules) before resorting to LLM hypothesis generation. Deterministic paths are preferred for explainability and cost. This is the single most important v1.5+ quality gate.

8. **Latency tiering:**
   - **Explicit retrieval** (`/ask`, synthesize=false): <250ms single-domain p95, <400ms all-domain p95.
   - **Explicit retrieval + synthesis** (`/ask`, synthesize=true): <3s p95 (all domains).
   - **Discovery / reasoning** (`/discover`, multi-hop, inferred): latency is strategy-dependent and measured per-strategy; no single SLA across all reasoning paths. Each strategy documents its expected budget.

9. **Inferred integrity** — every inferred relationship (a) carries a confidence score, (b) lists the explicit evidence it was derived from, (c) passes a deterministic validator before acceptance, (d) is stored SEPARATELY from the explicit graph (never corrupt source-of-truth data), and (e) is explainable (Why / How / Evidence / Confidence / Graph-path).

10. **Never sacrifice data integrity** — for recall, for speed, or for convenience. A rejected or low-confidence inferred edge is better than a high-confidence false one. The explicit knowledge graph remains the immutable source of truth.

## 3. Delivery Phases (7 days)

### Phase 1 — Core Performance (Days 1-2)

**Objective**: Fix retrieval pipeline bottlenecks so adding more domains doesn't compound slowness.

This phase implements 8 performance optimizations. Summary:

| # | Fix | Waste eliminated | Effort |
|---|-----|-----------------|--------|
| 1 | Pass precomputed `vec` to clinical_prose retriever → skip HTTP re-embed | 60-160ms | Medium |
| 2 | Merge `_idf_for_keyword` + `_concepts_for_keyword` into single Cypher | 30-60ms | Low |
| 3 | Remove redundant `_edge_boost` (subset of `_symptom_to_disorder`) | 20-40ms | Low |
| 4 | Module-level cached Neo4j driver + Qdrant client | 10-20ms | Low |
| 5 | Enable semantic cache for `synthesize=false` | 250ms (repeat queries) | Low |
| 6 | Improve `_names_batch` to include non-English fallback in one batch | 5-20ms | Low |
| 7 | Precompute SNOMED IDF lookup table (startup build, O(1) query-time) | 80-120ms per keyword query | Medium |
| 8 | Skip BGE reranker when `synthesize=false` and pool ≤ 10 (raw retrieval scores suffice) | 20-40ms | Low |

**Acceptance criteria**:
- `/ask (synthesize=false)` for "fever and rash" (single domain, cold) is < 250ms p95
- Multi-domain `domain=all` cold latency < 400ms p95
- Same contexts returned as before (ranking preserved where semantically equivalent; raw-score ordering acceptable for synthesize=false)
- Repeat queries hit cache regardless of `synthesize` flag (cache hit ~70ms end-to-end; the query embed runs before the cache lookup, so the floor is the Jina embed cost, not <10ms — the cache saves retrieval+rerank+synthesis, not embedding)
- No new errors in `logs/daemon_gpu.log`

**Deliverables**:
- Patched `clinical_prose/retrieve.py` (vec-accepting retrieve signature)
- Patched `domains/__init__.py` (forward vec in get_retriever lambda)
- Patched `snomed/retrieve.py` (merged Cypher, removed _edge_boost, cached driver, improved _names_batch, precomputed IDF lookup)
- Patched `serve_gpu.py` (cache gate fix, conditional reranker skip)
- Timing benchmark script: `scripts/bench_ask.py` (baseline vs optimized)

### Phase 3 — Enterprise Hardening (Days 4-6)

**Objective**: Make the system safe to expose publicly. No 500s, no silent failures,
no unrecoverable states. Observable, debuggable, and documented.

#### Day 4-5: Error Boundaries + Graceful Degradation

| Hardening | Where | What |
|-----------|-------|------|
| Neo4j unavailable | `snomed/retrieve.py`, all graph retrievers | Return `[]` + log warning; `/ask` continues with prose-only results |
| Qdrant unavailable | `clinical_prose/retrieve.py`, all dense retrievers | Return `[]` + log warning; `/ask` continues with graph-only results |
| Domain not found | `domain_loader.py` | Return `{"error": "unknown_domain", "available": [...]}` with 200 (not 500) |
| Empty collection | All retrievers | Return 0 contexts gracefully; no crash |
| E4B synthesis unavailable | `serve_gpu.py` `/ask` | Return partial result with `"answer": ""`, `"degraded": true`, `"notice": "synthesis unavailable"` |
| GLiNER extraction failure | `extractor_gliner.py`, sliding_window | Catch KeyError, continue with remaining windows; never abort full ingest |
| Concurrent write conflict | `ingest.py`, Neo4j MERGE | Use MERGE (idempotent), catch constraint violations, log and continue |

## 5b. Known Limitations (v1.0.0 — verified 2026-07-15)

These are documented, hardware/scoping-bound gaps. They are NOT code defects.

- **`default` alias.** The plan body says `default → healthcare` (P2.10) while
  §0/§2 say `default → snomed`. v1.0 implements `default: alias snomed` AND
  `healthcare: alias snomed` (identical retrieval). This is a doc inconsistency,
  not a functional gap.
- **`/domains` response shape.** P2.10 VERIFY expects a JSON list; the live
  endpoint returns `{"domains":[...], "default_domain":..., "count":...}`. The
  endpoint is correct; the VERIFY command in the plan was written for the wrong
  shape.
- **Neo4j/Qdrant-down graceful degradation** (P3.12) is verified by code-path
  simulation (per-domain try/except returns degraded + other domains), not by
  live `systemctl stop` of the shared infra on the reference host.

> **RESOLVED:** `synthesis=true` latency was p95≈10.4s (only healthcare's
> verbose differential). Fixed by tightening `prompts/clinical_dx.md` to a
> concise ≤5-candidate ranked differential (≤250 words). Post-fix 20x synthesis
> p95: healthcare 2.38s, enterprise 1.91s, legal 1.40s, fraud 1.18s — all
> under the 3s target, answers still a valid ranked differential.

### Official verified baseline (2026-07-15, RTX 3060 12GB; `skip_cache=true`, n=20/domain)

| Mode | healthcare | enterprise | legal | fraud | all-domain | errors |
|------|-----------|-----------|-------|-------|-----------|--------|

  `synthesize=false` regression, and variant-scoping (see CHANGELOG "Fixed").

## 11. PHASE 8 — Relationship Discovery (v2.0) ⭐

**Status: PLANNED — NOT IMPLEMENTED**
**This is the core differentiator.**

Polyglot-GraphRAG should discover implicit relationships that no individual
document explicitly states. This phase builds a candidate generator +
validator pair that produces an INFERRED knowledge graph, separate from the
explicit source-of-truth graph.

### Candidate Relationship Generator

Generate possible hidden connections from:
- [ ] **P8.1** Graph topology (transitive closure, shared neighbors, bridge nodes)
- [ ] **P8.2** Shared entities (two services sharing the same database/queue)
- [ ] **P8.3** Shared APIs (two services calling the same endpoint)
- [ ] **P8.4** Shared schema (two tables referencing the same foreign key)
- [ ] **P8.5** Ontology rules (domain-specific inference rules, e.g. SNOMED IS-A hierarchy)
- [ ] **P8.6** Embedding similarity (two descriptions that are semantically close)
- [ ] **P8.7** Temporal correlation (events that co-occur within a window)
- [ ] **P8.8** Dependency analysis (call graphs, data flows)
- [ ] **P8.9** Event flow (producer→broker→consumer chains)
- [ ] **P8.10** LLM hypothesis (as LAST resort — "deterministic before LLM")
  > VERIFY: each source enumerated above produces a Candidate with confidence,
  > reason, and supporting evidence. All sources together form the candidate pool.

### Relationship Validator

Never trust generated relationships immediately:
- [ ] **P8.11** Ontology constraints (does the inferred edge type make sense
  given the entity types?)
- [ ] **P8.12** Graph rules (does the edge contradict an existing explicit edge
  or form a cycle that violates domain rules?)
- [ ] **P8.13** Existing evidence check (is there direct evidence that supports
  or refutes the candidate?)
- [ ] **P8.14** Provenance check (are the source documents trustworthy?)
- [ ] **P8.15** Confidence threshold — gate on a configurable floor; candidates
  below the floor are rejected, not stored.
- [ ] **P8.16** Human approval (optional) — flag `needs_review` for edge cases.
  > VERIFY: a candidate pool of 50 → accepted ~15, rejected ~25, needs_review ~10,
  > each with a reason.

### Inferred Knowledge Graph

Stored SEPARATELY from the explicit graph:
- [ ] **P8.17** `inferred_depends_on` — A indirectly depends on C via B
- [ ] **P8.18** `inferred_impacts` — a change to A may impact C
- [ ] **P8.19** `inferred_related_to` — A and C share context but aren't directly linked
- [ ] **P8.20** `inferred_same_as` — A and C are likely the same entity
- [ ] **P8.21** `inferred_causes` — A may cause C (temporal/historical)
- [ ] **P8.22** `inferred_part_of` — A is likely a component of C
  > VERIFY: every inferred edge carries confidence, supporting_evidence,
  > strategy, created_at. Zero inferred edges in the explicit graph.

## 12. PHASE 9 — Reasoning Engine (v2.0) ⭐⭐⭐

**Status: PLANNED — NOT IMPLEMENTED**

Current flow: Question → Retrieve → Answer.
Future flow: Question → Reason → Discover → Explain → Answer.

### Multi-hop Reasoning Traversal

- [ ] **P9.1** Path expansion — from one entity, expand outward N hops through
  explicit AND inferred edges.
- [ ] **P9.2** Shortest path — find the shortest explanatory path between two entities.
- [ ] **P9.3** Weighted traversal — favor paths with higher confidence edges.
- [ ] **P9.4** Semantic traversal — follow edges whose type/description matches
  the query intent.
- [ ] **P9.5** Dependency traversal — follow depends_on / impacts edges.
- [ ] **P9.6** Impact traversal — "if A changes, what is affected?" — reverse
  dependency walk.
- [ ] **P9.7** Root cause traversal — "what caused X?" — walk backwards through
  causal edges.

### Reasoning Strategy Auto-Selection

Automatically choose a strategy based on the query intent:
- [ ] **P9.8** Dependency Analysis — "what depends on X?"
- [ ] **P9.9** Root Cause Analysis — "why did X happen?"
- [ ] **P9.10** Impact Analysis — "what would be affected if X changes?"
- [ ] **P9.11** Knowledge Discovery — "what hidden relationships exist?"
- [ ] **P9.12** Similarity Reasoning — "what is like X?"
- [ ] **P9.13** Ontology Reasoning — "what does X imply, given the ontology?"
- [ ] **P9.14** Temporal Reasoning — "what happened before/after X?"

### Explainable Reasoning

Every answer must explain:
- [ ] **P9.15** Why — the logical chain from question to conclusion
- [ ] **P9.16** How — the traversal path through the graph
- [ ] **P9.17** Evidence — which explicit edges/nodes support each hop
- [ ] **P9.18** Confidence — per-hop and cumulative confidence
- [ ] **P9.19** Graph path — the actual Cypher/algorithm that produced this result
- [ ] **P9.20** Sources — the original documents behind every piece of evidence

## 14. PHASE 11 — Knowledge Quality Engine (v2.0)

**Status: PLANNED — NOT IMPLEMENTED**

Detect problems in the knowledge graph automatically.

### Contradictions

- [ ] **P11.1** Conflicting facts — two edges of the same type but opposite
  direction or meaning.
- [ ] **P11.2** Version conflict — an entity described differently in two
  documents of different timestamps.
- [ ] **P11.3** Ontology conflict — an inferred edge that violates a domain rule.

### Missing Knowledge

Detect gaps:
- [ ] **P11.4** Missing nodes — entities referenced but not defined.
- [ ] **P11.5** Missing relationships — entities that should be connected
  (by domain rule) but aren't.
- [ ] **P11.6** Missing documentation — entities with no source document
  provenance.

### Stale Knowledge

- [ ] **P11.7** Outdated docs — documents older than a threshold with no updates.
- [ ] **P11.8** Deprecated APIs — APIs marked deprecated in specs but still
  linked as active.
- [ ] **P11.9** Dead code — code entities with zero inbound references.
- [ ] **P11.10** Unused entities — graph entities with degree = 0.

### Graph Health

- [ ] **P11.11** Orphan nodes (degree = 0, already partially covered by
  `graph_prune.py`).
- [ ] **P11.12** Duplicate nodes (near-identical name + embedding).
- [ ] **P11.13** Weak relationships (confidence < threshold).
- [ ] **P11.14** Low-confidence areas (subgraphs where average confidence
  is below threshold).

## Final North Star

Polyglot-GraphRAG is **not** a document retrieval system.

Polyglot-GraphRAG is **not** a knowledge graph builder.
