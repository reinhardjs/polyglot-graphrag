# Plan: v1.0 Requirement Alignment — `notice` field + CRAG scope

## Status: PROPOSED (no code changes made yet — user requested report-only)

## Context

A restated v1.0 enterprise-MVP requirement was reviewed against the live system
(2026-07-15). ~95% of it is already implemented and verified — it maps 1:1 onto
`plans/v1.0-enterprise-mvp-roadmap.md`. This plan captures the ONLY 3 open items
so they don't get lost.

The requirement's binding constraints (for reference):
1. ZERO ARCHITECTURAL DRIFT — no background daemons, inferred graphs, or
   multi-hop reasoning; explicit vector + graph retrieval only.
2. Latency — `/ask` < 250ms (single-domain, no synth), < 3s (with synth).
3. Graceful degradation — never a 500; return partial results with
   `"degraded": true` AND a `"notice"` explaining what was unavailable.
4. Provenance — every context carries `signal`, `concept_id`, `confidence`,
   `doc_id`, `chunk_idx`.
5. One request/response — `/ask` is the only endpoint a consumer needs.

## Already aligned (verified live — no work needed)

- Constraints #2, #4, #5: MET. Provenance confirmed present (all 5 fields);
  `/ask` is the single surface; latency baseline retrieval all-domain p95=89ms,
  synthesis all <3s.
- Core perf (singletons, vec bypass, merged Cypher, precomputed IDF, cache,
  reranker skip): DONE (P1.1–P1.8).
- 4 domains + cross-domain fan-out with `_domain` tags: DONE.
- Graceful degradation (per-domain try/except, unknown domain → 200, E4B fail →
  degraded, no 500s): DONE.
- Observability (`/health`, `/metrics`, `X-Request-ID`): DONE.

---

## A1 — Add `notice` field to all degraded responses  (constraint #3)

**Gap:** Degraded responses today carry `degraded` + `failed_domains` (and
`error`+`available` for unknown domain) but NO human-readable `notice` string,
which constraint #3 explicitly requires.

- [ ] **A1.1** Add a `notice: str` field to the `/ask` result dict. Populate it
  whenever `degraded` is true or a domain/backend was unavailable. Compose from
  `failed_domains` + E4B status, e.g.
  `"synthesis unavailable (E4B unreachable); returned retrieval-only"` or
  `"domain 'fraud' retriever failed; results exclude it"`.
  > VERIFY: force an E4B-down synth request → response has `degraded:true` AND a
  > non-empty `notice`. Healthy request → `notice` empty or absent.
- [ ] **A1.2** Additive only — never remove/rename existing fields (API contract
  rule). `notice` defaults to `""` on the happy path.
  > VERIFY: `grep '"notice"' serve_gpu.py` present; happy-path `/ask` unchanged
  > shape except the new optional field.

**Risk:** none — additive field, zero drift, no latency impact.

---

## A2 — Unify unknown-domain degraded semantics  (constraint #3)

**Gap:** Unknown domain returns `degraded:false` + `error:"unknown_domain"`.
The requirement frames unknown-domain as a degradation case (expects
`degraded:true` + `notice`).

- [ ] **A2.1** Decide + implement: unknown domain returns
  `degraded:true`, `error:"unknown_domain"`, `available:[...]`, and
  `notice:"unknown domain 'X'; available: [...]"`. Keep HTTP 200.
  > VERIFY: `curl /ask -d '{"domain":"nope"}'` → 200, `degraded:true`,
  > `notice` names the bad domain + lists available ones.

**Risk:** low — behavioural tweak on an already-200 path.

---

## A3 — Resolve CRAG vs "ZERO DRIFT"  (constraint #1) — DECISION NEEDED

**Conflict:** `crag_pipeline.py` is wired into `/ask` via a `crag:bool` flag
(default false). Its corrective pass REWRITES the query via E4B and re-retrieves
— arguably the "multi-hop reasoning" constraint #1 forbids. It does not run
unless `crag:true`, so it never touches the default path or latency, and it is
NOT part of the v1.0 roadmap success criteria.

Options (pick one — user decision):
- [ ] **A3-opt-a (keep):** Leave CRAG in place, off by default; document it as an
  opt-in EXPERIMENTAL extension explicitly outside the v1.0 explicit-retrieval
  contract. Lowest effort; small doc note.
- [ ] **A3-opt-b (hide):** Remove the `crag` field from the public `AskReq` (keep
  `crag_pipeline.py` on disk) so the v1.0 `/ask` surface is strictly
  explicit-retrieval-only. Reversible.
- [ ] **A3-opt-c (gate):** Move CRAG behind a config flag (`ENABLE_CRAG`,
  default false) so the surface only appears when an operator opts in.

> VERIFY (whichever chosen): default `/ask` has no reasoning/rewrite behaviour;
> the explicit vector+graph path is the only path a normal consumer can reach.

**Note:** CRAG is a natural stepping-stone toward the separate v2.0 reasoning
platform (see `plans/v2.0-reasoning-discovery-platform.md`). Recommend **A3-opt-c
(gate)** so it survives as a v2.0 seed without polluting the v1.0 contract.
