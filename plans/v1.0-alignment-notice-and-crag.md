## Status: EXECUTABLE — all decisions resolved, ready for `hermes /goal achieve`

Everything else (latency, provenance, one-endpoint-surface) is verified and

## CONSTRAINTS (read once, apply always)

  ./venv/bin/python serve_gpu.py > logs/daemon_gpu.log 2>&1 &`

  `"status":"ok"` before proceeding.
- Verify EVERY change live with `curl POST /ask` against the running daemon.
- NEVER use `sed -i` on config.yaml. Use write_file + cp or the patch tool.
- NEVER call `.half()` on CPU models.
- The daemon PID changes across restarts; always discover the live PID with
  `pgrep -f serve_gpu.py` — never assume a stale PID.

### DECISIONS (pre-resolved — no ambiguity)

- A3 (CRAG scope): Gate CRAG behind a config flag (`ENABLE_CRAG`, default
  `False`). Keep `crag_pipeline.py` on disk but do NOT expose the `crag` field
  on the public `AskReq` when the flag is off. This keeps v1.0
  explicit-retrieval-only by default while preserving CRAG as a v2.0 seed.
- A2 (unknown-domain semantics): Unknown domain MUST return `degraded:true`,
  `error:"unknown_domain"`, `available:[...]`, and a `notice` string naming
  the bad domain. HTTP 200, not 500. This unifies the unknown-domain path
  with the general degraded contract.

### A1 — Add `notice` field to all degraded responses

- [ ] **A1.1** Add a `notice` field to the `/ask` result dict (see §CONSTRAINTS:
  additive only). Populate from `_failed` + E4B status. Format:
  `"domain 'fraud' retriever failed; results exclude fraud"` or
  `"synthesis unavailable (E4B unreachable); returned retrieval-only results"`.
  Default to `""` on the non-degraded path.
  > VERIFY: force an E4B-down synth request (or use a domain whose
  > retriever throws) → response has `degraded:true` AND a non-empty `notice`.
  > VERIFY: healthy `/ask` (domain=snomed, synth=false) → `notice` is `""`
  > OR absent; all other fields unchanged from before.
  > VERIFY: `grep '"notice"' serve_gpu.py` returns a match on the result
  > assembly line.

---

### A2 — Unify unknown-domain degraded semantics

- [ ] **A2.1** Change the unknown-domain response to include:
  `degraded:true`, `error:"unknown_domain"`, `available:[...]`, and
  `notice:"unknown domain 'X'; available: [A, B, C, ...]"`. Keep HTTP 200.
  > VERIFY: `curl -s -X POST localhost:8000/ask -H 'Content-Type: application/json'
  > -d '{"query":"test","domain":"nonexistent","synthesize":false}' | python3 -c
  > "import sys,json; d=json.load(sys.stdin); print(d.get('degraded'),
  > d.get('notice','NO NOTICE FIELD'))"`
  > → prints `True` and a notice string containing `"nonexistent"` and the
  > available domain names.
  > VERIFY: HTTP status is 200 (not 500).

---

### A3 — Gate CRAG behind `ENABLE_CRAG` config flag

- [ ] **A3.1** Add `ENABLE_CRAG: bool = False` to `config.py`.
  > VERIFY: `grep "ENABLE_CRAG" config.py` returns a match.

- [ ] **A3.2** In `serve_gpu.py`: wrap the `crag` field on `AskReq` and the
  CRAG dispatch branch (`if req.crag:`) behind the config flag. When
  `ENABLE_CRAG` is `False`, the `crag` field is absent from `AskReq` (or
  silently ignored) and the CRAG branch is never reachable. When `True`,
  existing CRAG behaviour is preserved unchanged.
  > VERIFY: with `ENABLE_CRAG=False` (default), `grep "crag" serve_gpu.py`
  > shows the field is gated; `curl /ask -d '{"crag":true,...}'` either
  > ignores the field or returns a 422; the CRAG import + dispatch is
  > never executed on the normal `/ask` path.
  > VERIFY: set `ENABLE_CRAG=True`, restart daemon, pass `crag:true` in a
  > request → CRAG path runs and returns the `source:"crag"` response shape
  > (confirm it still works).

- [ ] **A3.3** Document: add a 2-sentence note in `serve_gpu.py` above the
  CRAG gate: `# CRAG is gated behind config.ENABLE_CRAG (default false).
  # Enable to opt into experimental query rewriting + corrective retrieval.`
  > VERIFY: `grep "ENABLE_CRAG" serve_gpu.py` returns 2+ hits (config read
  > + gate), plus the comment.
